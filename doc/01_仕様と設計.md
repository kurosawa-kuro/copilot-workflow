# Copilot ワークフロー 仕様と設計

> ワークフローの仕様・設計思想・タスク種別定義。LLM に全体像を伝える土台。
> 要件レベルの変更時に更新する。矛盾時は [`src/01_workflow-guide.md`](../src/01_workflow-guide.md) を正とする。

## 共通前提

- **目的**: GCP/MLOps チームが GitHub Copilot を使い、品質を保ちながら START（仕様書）→ GOAL（PR）まで進める手順を固定化する。
- **入力 / 出力**:
  - 入力 = 仕様書・既存コード・調査結果
  - 出力 = 作業計画書（Gate①提出）→ 実装コード一式（テスト込み・自己レビュー済、Gate②提出）→ PR
- **担当範囲**: ワークフロー定義とプロンプト雛形の設計・管理。モデル自体の性能改善・データサイエンス作業はスコープ外。
- **主要技術スタック**: GCP（Vertex AI / Vertex AI Pipeline / BigQuery / Feature Store / Cloud Run Jobs）/ Elastic Cloud（Elasticsearch）/ Terraform / Python
- **対象タスク種別**: Cloud Run Job（Pythonバッチ）/ Vertex AI Pipeline / Elastic Search / Terraform（FastAPI は例外的）
- **開発環境**: テキストエディタ + git。コードを持たないドキュメント専用リポジトリ。ビルド・lint・test コマンドは存在しない。
- **言語**: 日本語が canonical。コミットメッセージ・PR タイトルも日本語。

## 1. アーキテクチャ

設計対象は「アプリケーション」ではなく「開発作業の構造」である。作業の流れは「仕様把握と不足確認 → リーダー合意 → 実装 → リーダー承認」とする。

```text
仕様書を受け取る
        │
        ▼
[確認フェーズ] 実装しない
  1. 仕様書レビュー（矛盾・曖昧・未定義・境界条件の抽出）
  2. 既存コード照合 + 実在確認（関数/API/ライブラリを実物で裏取り）
  3. 不足情報リスト化（仕様/データ/API/テスト観点の欠落）
  4. 作業計画書作成（ゴール・成功定義・共通認識・実装方針・リスク等）
        │
        ▼ 作業計画書を提出
[Gate①] リーダー承認 ←────────────── 差し戻し → 4 へ
        │ 承認
        ▼
[実装フェーズ] ここからコードを書く
  5. テストコード作成（先にテスト）
  6. テスト仕様準拠チェック
  7. 実装コード作成（テストを満たす最小実装）
  8. レビュー＆修正（実在性 → 仕様逸脱 → セキュリティ → データ不整合 → 性能）
        │
        ▼ 実装コード一式（テスト込み・自己レビュー済）を提出
[Gate②] リーダー承認 ←────────────── 差し戻し → 7（または 5/4/1）へ
        │ 承認
        ▼
       PR
```

### 1.1 フェーズの責任境界

| フェーズ | 書いてよいもの | 書いてはいけないもの |
|---|---|---|
| 確認フェーズ | 作業計画書（方針・地図） | プロダクションコード・テストコード |
| 実装フェーズ | テストコード → 実装コード | 仕様書の再解釈（確認フェーズに戻る） |

### 1.2 Gate の通過条件

| Gate | 提出物 | 通過条件 |
|---|---|---|
| Gate① | 作業計画書 | ゴール・成功定義・共通認識が合意できる内容 / 実在確認済み / `[要検証]` が確定事項に紛れていない |
| Gate② | 実装コード一式 | テストが全件グリーン / 実在しない呼び出し（S 級）が 0 件 / 自己レビュー済み |

## 2. 機能仕様（8 ステップの定義）

| ステップ | 名称 | 入力 | 出力 | 禁止事項 |
|---|---|---|---|---|
| 1 | 仕様書レビュー | 仕様書 | 矛盾・曖昧・未定義・境界条件リスト | — |
| 2 | 既存コード照合 + 実在確認 | 仕様書・既存コード | 実在確認表（確認方法・結果・根拠セット） | AI に「ありますか？」と聞いて確認したことにする |
| 3 | 不足情報リスト化 | 仕様書・ステップ2結果 | 不足情報リスト（根拠・確認先付き） | — |
| 4 | 作業計画書作成 | ステップ1〜3結果 | 作業計画書（コードなし） | コードサンプルを含める |
| 5 | テストコード作成 | 作業計画書 | テストコード（実装なし） | 実装コードを先に書く |
| 6 | テスト仕様準拠チェック | テストコード・仕様 | 過不足指摘表 | — |
| 7 | 実装コード作成 | テストコード | 実装コード（テストを変えない） | テストを弱める修正 |
| 8 | レビュー＆修正 | 実装コード | Severity 付き指摘・修正済みコード | S/A 指摘を放置してゲート②に出す |

## 3. 設計上の決定事項

### 3.1 採用したアプローチと理由

- **2 ゲート制を採用**: 確認フェーズと実装フェーズを強制分離することで、仕様の読み違えによる手戻りを防ぐ。Gate① で合意した計画書があることで、Gate② のレビューが実装品質の確認に集中できる。
- **テストファーストを強制**: ステップ 5（テスト）→ ステップ 7（実装）の順を崩さない。テストを後から書くと「テストが実装に合わせて弱くなる」パターンを防ぐ。
- **実在確認（anti-hallucination）を必須化**: GCP API・ライブラリ・設定キーなど、Copilot が「ありそうな名前」を自信満々に生成する。grep / import / 公式ドキュメントによる実物確認を必須にすることで、存在しない呼び出しの混入を防ぐ。
- **タスク種別別の成功定義を作った**: Terraform はテストコードが書けない。Vertex AI Pipeline は pytest ではなくコンソールでの確認が主になる。種別によって確認方法が根本的に異なるため、種別ごとにテンプレートを用意した。
- **作業計画書にゴール・成功定義・共通認識を先置きした**: 実装方針だけでは「何が成功か」がリーダーとメンバーで異なる問題が発生した。先に合意することで差し戻しの原因を減らす。

### 3.2 採用しなかったアプローチ

1. **1 ゲート制** — 実装後のレビューだけでは、計画レベルの問題（スコープ誤解・設計ミス）が発見されるのが遅すぎる。
2. **コードファーストで確認** — 動くコードを見てから計画を立てる方法は、Copilot が生成したコードへの依存を生みやすく、仕様から外れた実装に気づきにくい。
3. **種別共通の成功定義** — 「pytest がグリーン」だけを成功定義にすると、Terraform や Vertex AI Pipeline のタスクで適用できない。

## 4. 非機能要件

- **整合性**: `src/01_workflow-guide.md` と `src/02_prompt-templates.md` のステップ番号・mermaid 図・プロンプト番号は常に一致させる。片方を変えたら必ずもう片方も同一変更で更新する。
- **鮮度**: 技術スタックやタスク種別が変わったら、`src/02_prompt-templates.md` の前提欄・成功定義テンプレートを更新する。
- **検証可能性**: 作業計画書に書く実在確認表は、実際に実行したコマンドの出力を根拠とする。AI の記憶での断定は禁止。
- **ドキュメント専用**: コードを持たない。ビルド / lint / test / デプロイコマンドは存在しない。
