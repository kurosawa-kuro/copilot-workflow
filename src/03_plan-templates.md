## 成功定義

<!-- タスク種別に合ったテーブルを残し、他を削除して埋める -->

### Cloud Run Job（Pythonバッチ）の場合

| 成功条件 | 確認方法 |
|---|---|
| ジョブが正常終了する | Cloud Run Job の exit code 0 + Cloud Logging でエラーなし |
| 処理件数がログで追跡できる | Cloud Logging に処理件数・完了メッセージあり |
| BigQuery / GCS への出力が仕様と一致する | 出力テーブルの行数・値・スキーマを確認（ログ確認だけでは不十分） |
| 再実行しても二重書き込みが起きない | 2回実行して行数・レコードが変わらないことを確認 |

> ログ確認は「処理が完了したか」の確認。「正しい結果が出たか」は作成物（BQ テーブル / GCS ファイル / Feature Store 値）のチェックが必要。両方を成功定義に含める。

### Vertex AI Pipeline の場合（テストコードは限定的）

| 成功条件 | 確認方法 |
|---|---|
| 全コンポーネントが正常完了する | Vertex AI コンソールでパイプライン実行ステータスを確認 |
| 各コンポーネントの出力アーティファクトが正しい | GCS / Vertex AI Experiments で中間出力を確認 |
| Feature Store に正しい値が書き込まれている | Feature Store API または BQ エクスポートで値を確認 |
| BQ 出力テーブルが仕様の形式・件数と一致する | クエリで行数・値サンプルを確認 |

### Elastic Search の場合（テストコードは書かない）

| 成功条件 | 確認方法 |
|---|---|
| インデックスのドキュメント数が期待通り | `_count` API で確認 |
| クエリのレスポンスが仕様の形式と一致する | 代表的なクエリを実行してレスポンスを目視確認 |
| 既存インデックスへの意図しない影響がない | 別インデックスのドキュメント数・クエリ動作を確認 |

### Terraform の場合（テストコードは書かない）

| 成功条件 | 確認方法 |
|---|---|
| 構文エラーがない | `terraform validate` が通る |
| 変更内容が意図通り | `terraform plan` の差分を目視確認 |
| 既存 GCP リソースへの意図しない変更がない | plan 出力の destroy/change 件数を確認 |
| apply 後にリソースが期待状態 | Google Cloud コンソールまたは `gcloud` コマンドで確認 |
