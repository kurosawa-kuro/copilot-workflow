# Copilot プロンプト例集（Python / バッチ）

[Copilot 開発運用ガイド](workflow-guide.md) の各ステップで Copilot に渡すプロンプト雛形。

**前提**
- 言語: Python
- 種別: 非バックエンド（Web API / サーバーではない）
- 主用途: バッチ処理（定期実行・データ加工・ファイル/DB入出力・ETL）
- 重視点: 冪等性 / 再実行安全性 / 大量データのメモリ・チャンク処理 / 入力データの異常系 / ログと失敗時の追跡

使い方: `<>` を埋めて貼り付ける。確認フェーズではプロダクションコードもテストコードも書かせない。「実装サンプルメモ」（任意・自分用／提出不要）を作る場合も、実装ではなく自分用の方向性確認に留める。

---

## 確認フェーズ（実装しない＝プロダクションコードは書かせない）

### 1. 仕様書レビュー

```
あなたはレビュアーです。コードは書かないでください。
以下の仕様書を読み、Pythonバッチとして実装する前提で次を抽出してください。
- 矛盾している記述
- 曖昧な語句（「適宜」「必要に応じて」等）
- 未定義の条件（入力が空/不正/重複のときの挙動など）
- 境界条件の不足（件数0、上限超過、日付境界、文字コード、欠損値）
バッチ特有の観点（再実行時の扱い、途中失敗時の扱い、実行タイミング）も漏れなく指摘してください。

# 仕様書
<仕様書を貼る>
```

### 2. 既存コード照合

```
コードは書かないでください。調査のみ。
以下の既存バッチコードと仕様を照合し、次を出してください。
- 仕様を満たすために実装可能か（可否と理由）
- 既存の処理フロー/設定/共通関数と衝突する箇所
- 既存の入出力形式（ファイルパス、スキーマ、文字コード、区切り）との不一致
- 流用できる既存関数・ユーティリティ（ファイル名と関数名で）
- あなたが挙げる関数 / メソッド / ライブラリ / API / 設定キー / DBカラムには出典を付ける（既存コード=ファイルパスと関数名 / 標準・公式=モジュール名と可能ならdoc URL / 確証なし=行頭に [要検証]）。出典を書けないものを「使える」と書かない

# 仕様
<仕様>
# 既存コード
<該当ファイル/関数を貼る>
```

#### 2-検証. 実在確認をAIに実行させる（依頼文・証跡を出させる）

検証作業そのものはAIにやらせてよい。ただし**「判断」させず、検証コマンドを走らせて証跡（確認方法・結果・根拠）を出させる**。`確認しました。存在します。` は不可。`rg "calculateScore" src/` → `src/domain/scoring.py に定義あり` のように出力を伴うこと。

```
以下の仕様対応に関連して、あなたが提案した関数 / API / ライブラリ / 設定キー / DBカラムについて、実在確認を行ってください。

重要：
- あなたの知識で「存在する」と判断しないでください
- 必ず grep / rg / import / package定義（pyproject.toml）/ schema / migration / 公式doc / test実行 など、実物確認に基づいてください
- 確認できないものは「未確認」または「存在確認できず」としてください
- 存在しないものを既存前提にしないでください

出力形式（確認方法・結果・根拠を必ずセットで）：

| 対象 | 種別 | 確認方法 | 結果 | 根拠 |
|---|---|---|---|---|
| calculateScore | 関数 | `rg "calculateScore" src/` | 既存あり | `src/domain/scoring.py` |
| pandas | ライブラリ | `python -c "import pandas; print(pandas.__version__)"` | 依存あり | `pyproject.toml` |
| user.score | DBカラム | migration確認 | 未確認 | 該当migrationなし |

# 仕様 / 提案した依存一覧
<対象を貼る>
```

> 未確認（`[要検証]`）のものは既存前提にせず、ゼロになるまで提出①に出さない。

#### 2-検証の詳細. 関門・確認方法・禁止（柱③の参照元）

[workflow-guide.md](workflow-guide.md) の柱③「実在確認」の詳細はここに集約する。

**関門（各ゲートの前に通す）**

| 関門 | やること | 満たすべき条件 |
|---|---|---|
| ステップ2 → ゲート①前 | 外部依存を全リスト化し実物で確認、証跡を計画書に添付 | `[要検証]` が 0件 |
| ステップ8 → ゲート②前 | import / test / 型チェックが実際に通る | 存在しない呼び出し（S級）が 0件 |

**確認方法（種別ごと・実物で裏取り。実行はAIエージェントに叩かせてよい）**

- 関数・型・クラス: `rg` / `grep` / IDE検索
- import可能性: `python -c "import xxx"` / `node -e "require('xxx')"` / `go list`
- ライブラリ: `package.json` / `pyproject.toml` / `go.mod` / lockファイル
- 設定キー: `.env.example` / config / Secret Manager / CI設定
- DBカラム: migration / schema / ORM定義 / 実DBの `\d`・`describe`
- 外部API: 公式doc または既存clientの実装
- 挙動: 最小テスト / dry-run / 型チェック / lint / unit test

**禁止**

- AIに「ありますか？」と聞いて確認したことにする
- AIの説明だけで既存要素として扱う
- grep / import 確認なしに作業計画書へ確定事項として書く
- モックで幻覚を隠す（実在するインターフェースにだけモックを作る）

> 出力形式（`対象 / 種別 / 確認方法 / 結果 / 根拠`）は上の「2-検証」プロンプトの表をそのまま使う。

### 3. 不足情報リスト化

```
コードは書かないでください。
仕様を実装するために足りない情報を、根拠付きで列挙してください。
カテゴリ: 仕様 / 入力データ（サンプル・スキーマ） / 出力要件 / 外部依存（DB・API・ファイル置き場） / テスト観点
各項目に「なぜ必要か」を1行添えてください。確認すべき相手（クライアント/リーダー）も明記。
各外部依存（DB・API・ライブラリ・設定キー）に出典または [要検証] を付ける（実在確認の対象になる）。
```

### 4. 作業計画書作成

```
コードは書かないでください。作業計画書のみ作成してください。
コードサンプルは含めないこと（方針・地図に徹する。コードで方向性を確認したい時は、自分用の実装サンプルメモ＝任意・提出不要で扱う）。
以下を Markdown で出力:
- 実装方針（処理の流れを箇条書き）
- 変更/新規対象ファイル
- 影響範囲（既存バッチ・共通処理・出力先）
- テスト方針（pytest 想定、正常/異常/境界）
- リスク（冪等性・大量データ・文字コード・タイムゾーン・部分失敗）
- 確認事項
- コミット計画（1コミット=1論理変更、テスト→実装の順、メッセージ方針）

# 仕様 / 調査結果
<3までの結果を貼る>
```

#### 4-任意. 実装サンプルメモ（自分用サポート資料・リーダー提出不要）

作業に慣れるまで、自分が実装の方向性を確認するために使う。リーダーには提出しない（ゲート①に出すのは作業計画書だけ）。実装フェーズ（7. 実装コード作成）の手元参考にもする。

```
※これは実装作業ではなく、自分用に方向性を確認するための抜粋スケッチ。動く完成実装は書かない。
実装サンプルメモを作成してください。これから書く新規・変更コードの重要箇所だけを抜粋サンプルで示します。
- 目指す関数/クラスのシグネチャと責務（入出力の型）
- 処理の要となる箇所（チャンク処理・冪等性・異常系の分岐など）の短い抜粋
- 既存コードとの接続点（呼び出し関係）を一言
制約: 重要箇所の抜粋のみ。完成コード全文や既存コードの長文貼り付けは禁止。作業計画書の内容は繰り返さない。
```

> ── ゲート①: 作業計画書を提出 → リーダー承認（差し戻しなら4へ）──

## 実装フェーズ（ここからコードを書く）

### 5. テストコード作成（先にテスト）

```
承認済みの作業計画書に基づき、まずテストだけ書いてください（実装コードはまだ書かない）。
- pytest 形式
- 正常系 / 異常系（不正・欠損・空入力）/ 境界（0件、上限、日付境界、文字コード）
- バッチ観点: 同じ入力での再実行が同結果になる（冪等性）、途中失敗後の再実行
- 入出力は一時ディレクトリ（tmp_path）やサンプルデータを使い、外部依存はモック/フィクスチャ化
テスト関数名は日本語コメントで意図を添えてください。

# 作業計画書
<計画書を貼る>
```

### 6. テスト仕様準拠チェック

```
コードは書かないでください。
書いたテストが作業計画書/仕様を正しく表現しているか確認してください。
- 仕様にない挙動を勝手にテスト化していないか
- 抜けている観点（異常・境界・冪等性）はないか
過不足を表で指摘してください。
```

### 7. 実装コード作成

```
テストを満たす最小実装を書いてください（テストを変更しないこと）。
- Python バッチとして: main()/エントリポイント、引数or設定ファイル、ログ出力、終了コード
- 大量データはチャンク/イテレータで処理（全件メモリ展開を避ける）
- 失敗時に途中状態を残さない or 再実行で復旧できる構造
- 例外は握りつぶさずログに残す
型ヒントと簡潔な docstring を付けてください。
```

### 8. レビュー＆修正

```
このバッチコードを、致命度の高い順にレビューしてください。
順序: 実在性（呼び出す関数/引数/ライブラリ/設定が実在しシグネチャが正しいか）→ 仕様逸脱 → 既存挙動破壊 → セキュリティ（認証情報・パス・任意コード実行）→ データ不整合（冪等性・部分失敗）→ パフォーマンス（メモリ・I/O）→ 保守性
実在性は import実行とテスト実行の結果を添えること（推測で「実在する」と書かない）。
各指摘に Severity（S/A/B/C）と修正案を付けてください。S/A から対応します。

# 対象コード
<コードを貼る>
```

> ── ゲート②: 実装コード一式（テスト込み・自己レビュー済）を提出 → リーダー承認 → PR ──

## 補足: バッチでよく効く追加プロンプト

```
このバッチに「再実行安全性」の観点で穴がないか確認してください。
（途中で落ちた場合、もう一度実行して二重登録・二重出力・データ破損が起きないか）
```

```
大量データ（<想定件数>件）でメモリが膨らまないか確認し、必要ならチャンク処理に書き換えてください。
```

```
入力ファイルの文字コード/区切り/改行/欠損値が想定外だった場合に、
落とすべきか・スキップしてログするか・止めるかを仕様に照らして整理してください。
```

```
生成コードが実在しない関数 / 引数 / ライブラリ / 設定キー / DBカラムを使っていないか、
import実行とテスト実行で確認してください。モックは実在するインターフェースに対してだけ作り、
外部依存は最低1回スモークで実物を叩いて確認（モックが幻覚を隠さないように）。
```
