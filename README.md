# copilot-workflow

GitHub Copilot を使った開発作業を、メンバーが **START（仕様書）→ GOAL（PR）** まで迷わず進めるための**運用フローとプロンプト雛形**をまとめたドキュメント集。コードは持たず、成果物は Markdown のみ。

## 何のためのリポジトリか

仕様書を受け取ったメンバーが、Copilot を使いながら品質を落とさず実装まで到達できるよう、手順を固定化する。途中で **2 回リーダーに提出して承認を待つ**（2 ゲート制）ことで、確認漏れや先走った実装を防ぐ。

鉄則は **確認フェーズではコードを書かない**（プロダクションコードもテストコードも）こと、そして **テストファースト**（テスト作成 → 実装）。

## ワークフロー全体像

```
▶ START（仕様書）
   └ 確認フェーズ（実装しない）: 1.仕様書レビュー → 2.既存コード照合 → 3.不足情報リスト化 → 4.作業計画書作成
        └ ★ゲート①: 作業計画書を提出 → リーダー承認
   └ 実装フェーズ（ここからコードを書く）: 5.テスト作成 → 6.テスト準拠チェック → 7.実装 → 8.レビュー＆修正
        └ ★ゲート②: 実装コード一式（テスト込み・自己レビュー済）を提出 → リーダー承認
🏁 GOAL（PR）
```

差し戻しはゲート②ならステップ7が原則。原因に応じて前工程（テスト不足→5 / 計画不備→4 / 仕様不足→1）へ戻る。

## ドキュメント早見

- **仕様書** … 何を満たすか
- **作業計画書** … どう進めるか（コードは原則書かない／Claude Code・Codex・Cursor の plan mode 相当）【必須提出物】
- **実装サンプルメモ** … 重要箇所の抜粋サンプル【任意・自分用／リーダー提出不要】

## 前提（プロンプト集）

[src/02_copilot-prompts.md](src/02_copilot-prompts.md) は **GCP（Vertex AI / BigQuery / Cloud Run Jobs / Elastic Cloud）/ Terraform を主軸とした MLOps 基盤**を想定し、冪等性 / 再実行安全性 / GCP API クォータ・コスト管理 / パイプライン失敗リカバリ / ログ追跡を重視する。
